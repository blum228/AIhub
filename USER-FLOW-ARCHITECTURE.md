# USER FLOW ARCHITECTURE

> Фундаментальный редизайн с точки зрения пользователя

---

## ТЕКУЩАЯ ПРОБЛЕМА

Сайт построен от **структуры данных**, а не от **задачи пользователя**.

Есть модели → показываем модели.
Есть теги → показываем теги.
Есть топы → показываем топы.

Это **каталожное мышление**. Пользователю плевать на структуру. Ему нужен ОТВЕТ.

---

## КТО ПРИХОДИТ И ЗАЧЕМ

### Пользователь A: "Хочу попробовать"
- Первый раз слышит про AI-девушек
- Хочет понять что это и попробовать бесплатно
- **Нужен:** быстрый старт без риска

### Пользователь B: "Хочу конкретное"
- Знает чего хочет: "с голосом", "NSFW", "на русском"
- **Нужен:** фильтр → результат → выбор

### Пользователь C: "Хочу лучшее"
- Готов платить, хочет топовый сервис
- **Нужен:** рейтинг, сравнение, рекомендация

### Пользователь D: "Сравниваю A и B"
- Уже знает 2 сервиса, выбирает между ними
- **Нужен:** прямое сравнение

## ГЛАВНАЯ СТРАНИЦА — ТОЧКА ПРИНЯТИЯ РЕШЕНИЯ

```

Это витрина. Пользователь должен сам понять что делать.

      ↓
Три пути:
┌─────────────────┬─────────────────┬─────────────────┐
│ 🆓 ПОПРОБОВАТЬ  │ 🔍 НАЙТИ ПО    │ ⭐ ЛУЧШИЕ      │

```

Это **decision tree**. Пользователь выбирает путь, а не блуждает.

## ПЛАН РЕАЛИЗАЦИИ

### Фаза 1: Главная как Decision Tree
1. Три пути: Попробовать / Найти / Лучшие
2. Убрать превью карточек (они в каталоге)
3. Добавить социальное доказательство (N сервисов, N обзоров)

### Фаза 2: Каталог как инструмент
1. Фильтры как вопросы, не категории
2. Группировка: Цена / Фичи / Платформа
3. Счётчик результатов
4. Сортировка: По рейтингу / По цене

### Фаза 3: Карточка как ответ
1. Главная фича (не description)
2. Ключевые теги (3-4 максимум)
3. Два CTA: Подробнее + Попробовать

### Фаза 4: Страница модели как убеждение
1. Позиционирование (одна фраза)
2. Для кого / Не для кого
3. Альтернативы
4. Один финальный CTA

### Фаза 5: Топы как готовые решения
1. Короткий список (3-5 позиций)
2. Одна рекомендация
3. Прямые CTA
